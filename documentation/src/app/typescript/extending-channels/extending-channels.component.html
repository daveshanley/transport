Talking to peers across the fabric.

Extending channels is the concept of being able to decide which channels you extend to a broker, and how you map them to destinations on those brokers.

Galactic Channels

When Transport was being developed, it was originally had a code name of 'BifrÃ¶st'. The term 'Galactic' Channel, stems from the idea that you could extend a channel to any other destination, on any compatible broker.

Essentially, marking a channel as galactic extends it to a broker destination, marking it as local boxes it to the local application only.

Marking a channel as Galactic

Use the bus.markChannelAsGalactic method. You can supply a broker identity, if you're using multiple brokers, to make sure you don't extend channels to all brokers.

Private Channels (Topics)

If you're using the Java or Golang version of Transport as your backend/service broker, you can use the optional isPrivate argument and set it to true. This will use a queue instead of a topic to communicate with your backend service. This is important if you don't want other clients or components in your applications to pick up this traffic.

Using this.fabric.generateFabricRequest?

This method creates a fabric ready request, capable of being routed across to services that are listening on remote / galactic channels, and that are using the Transport Java or Golang. You supply a command and an optional payload to send.

Can I message over galactic channels without using this.fabric.generateFabricRequest?

Yes. You can send what ever you like.
